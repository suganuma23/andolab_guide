ゼミ発表のコツ

ゼミでは先生から厳しいフィードバックをもらうことがあります。それは成長のチャンスです。以下のポイントを押さえておくと、発表の質がぐっと上がります。

  • 「なぜこれを勉強するのか」を最初に説明する — 動機づけがないと聴衆は置いていかれます。例えばハフマンコードなら「データ量を少なくしたい」という動機があります。それを先に伝えましょう。
  • スライドは統一感を大切に — フォント、サイズ、図の位置やサイズなど、無意味に変更しないこと。見た目の一貫性が伝わりやすさに直結します。
  • 文字は限界まで減らし、ビジュアルで表現する — スライドに文字を詰め込むと聴衆は読めません。図や表で直感的に伝えましょう。
  • 矢印は「変化」と「因果関係」に使う — 矢印の意味を統一すると、論理の流れが明確になります。
  • 一夜漬けで発表しない — 自分で「絶対正しい」と思ったことも、翌日になると間違いに気づくものです。余裕を持って準備しましょう。
  • 事前に先輩にチェックしてもらう — 自分では気づけないミスや論理の飛躍を指摘してもらえます。
  • 発表内容は必ず自分で確認する — 間違ったことを堂々と語ると後で辛くなります。ChatGPTの出力をそのまま使わず、自分の理解を通すこと。
  • 指摘を受けている最中は修正しない — まず聞くことに徹しましょう。全部聞き終えてから修正する方が正確です。
  • 感謝の気持ちを持って臨む — 発表の場は自分の成長の機会です。面倒と思わず、前向きに取り組みましょう。

先輩の体験談

発表後に自己嫌悪になるのは誰でもあります。でも数時間すれば落ち着きます。深呼吸して、次に活かすことだけ考えましょう。完璧な発表をする人はいません。回数を重ねるごとに確実に上手くなります。

論文の読み方

  • 論文は「間違っている」と思って読む — 正しいと信じて読むと疑問を見逃します。背理法の精神で、批判的に読みましょう。
  • 読み飛ばした部分が重要だったりする — 難しい部分こそ本質が書いてあることが多いです。
  • 分からないことが芋づる式に出てくるのは普通 — 焦らず一つずつ潰していきましょう。それが学びです。
  • 論文検索には MathSciNet と Google Scholar を使う — この2つを使いこなせれば、関連研究は大体見つかります。
  • 手書きで書いてからTeXにする方が楽 — 証明や数式は、まず紙に書いて写真を撮り、それを見ながらTeXに起こすと効率的です。

プログラミングの心得

  • リファクタリングに終わりはない — プログラムを見直すたびに改善点が見つかります。謙虚に、継続的に直していきましょう。
  • 関数の共通部分は必ずサブルーチン化する — コードの重複は保守性の敵です。
  • 同じfor文でできることは詰め込む — ループの分割は非効率。まとめられる処理はまとめましょう。
  • 無駄な処理・変数・引数は削除する — シンプルなコードが最も強いコードです。
  • 条件分岐はブロックが小さくなるように書く — 早期リターンなどを活用しましょう。
  • 命名規則を統一する — 関数名は「動詞_名詞」、変数名にアンダースコアを多用しない、など。
  • なるべくコードは短くする — ただし可読性とのバランスも大切です。

C言語の注意点

  • サブ関数内で malloc/free を繰り返すとフラグメンテーションでフリーズする可能性があります。メイン関数内で確保して使い回しましょう。
  • malloc したら必ず free しましょう。

効果的な学習法

研究室生活では、自分で新しい技術やツールを学ぶ場面が多くあります。以下は先輩が実践して効果があった学習の流れです。

  1. ChatGPTでやり方・理由・不明点を聞く — ここは「不確実でもいい場所」。大まかな方向性をつかむのに使います。
  2. 実際に試す — 環境・バージョン・自分の手で確認すること。机上の知識だけでは身につきません。
  3. うまくいったこと・腑に落ちたことだけをノートに残す — ノートは「検証済みの事実の集合」にしましょう。

学習に対する姿勢

  • 考える前に手を動かす — 計画ばかり立てても始まりません。まず手を動かして、やりながら修正していく方が結果的に早いです。
  • とにかく時間をかける — 「やり方」を調べるより、まず長時間取り組んでみることが一番の近道です。
  • 分からなければ人に聞く — 余計な寄り道をしなくなります。迷ったらすぐに先輩や先生に聞きましょう。一人で抱え込むのが一番非効率です。
  • 学んだことは資料にまとめておく — 後から見返せるだけでなく、人に教える時にも役立ちます。
  • 何度も見直す — レポートでもプログラムでも、見返すたびにミスや改善点が見つかります。提出前には必ず時間を置いてから確認しましょう。

開発環境・ツールの知識

研究室では Linux (Ubuntu)、SSH接続、Git、tmux などを日常的に使います。最初は戸惑いますが、早めに慣れておくと後が楽です。

Git

  • HTTPS接続ではなく SSH接続 を使いましょう。公開鍵・秘密鍵を設定すれば、パスワード入力なしで操作できます。
  • ~/.ssh/config にホスト設定を書いておくと接続が短縮できます。
  • ブランチの正体はポインタ。コミット履歴上のあるコミットを指しているだけです。この仕組みを理解すると、merge や rebase の意味がわかります。
  • merge は安全（共有ブランチ向き）、rebase は履歴がきれいになるが共有後は危険（個人ブランチ向き）。
  • git pull = git fetch + git merge です。

tmux

  • SSH接続が切れてもプログラムを実行し続けられるツールです。長時間の計算実行には必須。
  • exit せずに detach が正解。セッションを残したまま離脱しましょう。
  • ただし tmux は「SSH切断に強い」だけ。ローカルPCをスリープしても処理は止まります。長時間処理は必ずサーバ上で実行すること。

VSCode

  • Remote-SSH 拡張機能を使えば、VSCode からサーバに直接接続して編集できます。
  • Tailscale を使えば VPN 経由でどこからでもSSH接続可能です。
  • ショートカットキーをカスタマイズすると作業効率が大幅に上がります（keybinding.json）。

その他

  • ホームディレクトリ (~) に各種設定ファイルがあります（.bashrc, .gitconfig など）。
  • ドットファイル（. で始まるファイル・フォルダ）は隠しファイルです。ls -a で表示できます。
  • ~/.ssh に公開鍵・秘密鍵を置き、config にサーバ設定を書きます。
  • シェルのエイリアスは ~/.bashrc で設定できます。よく使うコマンドは短縮しましょう。

研究室生活の心得

日常について

  • みんながいる時間帯に研究室にいること — コミュニケーションの機会を逃さないように。
  • 結果を出すことが大事 — 研究室に長時間いることよりも、成果を出すことに集中しましょう。
  • 先生には敬語を忘れずに — 砕けた言い方は避けましょう。「マジですか」ではなく丁寧な言葉遣いを。
  • 飲み会は参加しておくと良い — 意外な共通点が見つかったりして、研究室の人間関係が深まります。

グループワークについて

  • 自分が大枠を作って、他の人には作業を任せる — 全体設計は自分でやり、具体的な作業を分担すると効率的です。
  • 成果物は他の人と共有してフィードバックをもらう — 自分では気づけないミスが必ずあります。
  • 提出前には必ず寝かせてから見返す — 時間を置くと客観的に見られるようになります。

メンタルについて

  • 不安や恐怖に囚われず、とにかく行動する — 行動だけが未来を変えます。悩んでいる時間はもったいない。
  • 怖いことの方が大事なことが多い — 卒論発表も、終わってみれば「何を恐れていたのか」と思うものです。
  • 発表後の自己嫌悪は誰でもある — 数時間すれば落ち着きます。深呼吸して、次に活かすことだけを考えましょう。
  • 迷ったら「やる」か「やらない」かをすぐ決める — 決断を先延ばしにするのが一番良くありません。やらないならやらない、やるならやる。
  • 完璧主義にならない — ハードルを下げて、まず始めることが大切です。

最後に

研究室生活は大変なこともありますが、振り返ると学びの多い時間でした。最初は何もわからなくて当然です。分からないことは恥ずかしがらずに聞いて、手を動かして、失敗して、そこから学んでください。

大事なことを5つだけ：

  1. 考える前にまず手を動かす
  2. 分からなければすぐ人に聞く
  3. 何度も見直す
  4. 決断を先延ばしにしない
  5. 感謝の気持ちを忘れない

みなさんの研究室生活が実りあるものになることを願っています。頑張ってください！
