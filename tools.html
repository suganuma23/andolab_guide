Git / GitHub

ソースコードはGitで管理する。HTTPS接続ではなくSSH接続を使う。

GitHubへのSSH接続

  1. 公開鍵と秘密鍵を作る（名前はGitHub用だとわかるようにする）
  2. 公開鍵をGitHubにアップロードする
  3. ~/.ssh/config に設定を記述する
  4. 接続テスト: ssh -T git@github.com
Host github.com
    HostName github.com
    User git
    IdentitiesOnly yes
    IdentityFile ~/.ssh/id_ed25519_github

ブランチの仕組み

ブランチの正体はポインタ。コミット履歴上のあるコミットを指しているだけ。

  • git fetch — リモートのコミット履歴をローカルにアップデート
  • git merge origin/<ブランチ名> — その情報を今いるブランチに取り込む
  • git pull = git fetch + git merge

merge と rebase の違い

履歴分岐が残る（安全）一直線になる
向いてる用途共有ブランチ個人ブランチ

VSCode

TailscaleでSSHする方法

  1. 各PCでTailscaleをインストールし、同じアカウントでログイン
  2. Tailscaleを起動する（Ubuntuでは sudo tailscale up）
  3. SSHしたいサーバのVPN上のIPアドレスをコピーして接続

公開鍵認証でSSHを楽にする

ssh andolab だけで接続できるようにする。

  1. 公開鍵・秘密鍵を作る
  2. 公開鍵をサーバーに送信し authorized_keys に追加
  3. ~/.ssh/config に設定を記述

tmux

1つのターミナル上で複数の仮想端末を管理できるツール。SSH接続が切れてもプログラムを実行し続けられる。

3階層構造

概念説明
セッション作業単位の一番大きな枠。SSHを切断しても残る
ウィンドウタブのようなもの。用途ごとに分ける
ペインウィンドウ内の分割画面

長時間処理の鉄則

tmuxは「SSH切断に強い」だけ。ローカルPCをスリープしても処理は止まる。
長時間処理は必ず「スリープしないマシン（サーバ）」上で使うこと。
exitせずdetachが正解。

プログラミングTips

心得

  • リファクタリングに終わりはない。見直す度に改善していく
  • 関数の共通部分は必ずサブルーチン化する
  • 無駄な処理、変数、引数は削除する
  • 条件分岐はブロックが小さくなるようにする

命名規則

  • 関数名は基本「動詞_名詞」
  • 変数名にアンダースコアは使わない（関数名と紛らわしいため）。変数はキャメルケース

C言語の注意点

  • サブ関数内で malloc/free すると断片化が起きる場合がある。メイン関数内で確保して使い回す
  • malloc したら free を忘れずに
