Git / GitHub

TeXの文章やプログラムのバージョン管理にはGitが有用です。ブランチを使えば異なるバージョンを並行して管理でき、任意のコミットに戻したり差分を確認したりすることも容易です。GitHubをクラウドリポジトリとして活用すれば、バックアップになるだけでなく、複数のPC間でのファイル共有・同期にも役立ちます。

以下は、初心者がゼロからGitHubを使えるようになるためのチュートリアルです。

1. Gitのインストール

Windowsの場合は git-scm.com からインストーラをダウンロードしてインストールします。Ubuntu / WSLの場合は以下のコマンドでインストールできます。

sudo apt update
sudo apt install git

インストールできたか確認:

git --version

2. 初期設定

Gitを使う前に、名前とメールアドレスを設定します。これはコミット履歴に記録されます。

git config --global user.name "あなたの名前"
git config --global user.email "your-email@example.com"

設定を確認:

git config --list

~/.gitconfig にエイリアスやアカウントの設定が書かれます。例えば git ac で add と commit を同時にできるエイリアスを追加することも可能です。

3. GitHubアカウントの作成

github.com でアカウントを作成してください。

4. SSH鍵の設定（重要）

GitHubとの接続にはHTTPS接続とSSH接続がありますが、SSH接続を使います。SSH鍵を設定すると、毎回パスワードを入力せずにGitHubとやり取りできます。

Step 1: SSH鍵ペアを作成する

ssh-keygen -t ed25519 -C "your-email@example.com" -f ~/.ssh/id_ed25519_github

パスフレーズを聞かれますが、空でもOKです。~/.ssh/ に以下の2ファイルが作られます:

  • id_ed25519_github — 秘密鍵（絶対に他人に見せない）
  • id_ed25519_github.pub — 公開鍵（GitHubにアップロードする）

Step 2: 公開鍵をGitHubに登録する

まず公開鍵の内容をコピーします:

cat ~/.ssh/id_ed25519_github.pub

表示された文字列を全てコピーし、GitHubの Settings → SSH and GPG keys → New SSH key に貼り付けて保存します。

Step 3: SSH設定ファイルを書く

~/.ssh/config に以下を追記します（ファイルがなければ新規作成）:

Host github.com
    HostName github.com
    User git
    IdentitiesOnly yes
    IdentityFile ~/.ssh/id_ed25519_github

Step 4: 接続テスト

ssh -T git@github.com

Hi ユーザー名! You've successfully authenticated... と表示されれば成功です。

5. リポジトリの作成

新しいリポジトリをGitHubで作成する場合

  1. GitHubで「New repository」をクリックし、リポジトリ名を入力して作成
  2. ローカルで以下を実行:
# プロジェクトフォルダに移動
cd ~/my-project

# Gitリポジトリを初期化
git init

# 最初のファイルを追加してコミット
git add .
git commit -m "Initial commit"

# リモートリポジトリを登録（SSHのURLを使う）
git remote add origin git@github.com:ユーザー名/リポジトリ名.git

# プッシュ
git push -u origin main

既存のリポジトリをクローンする場合

git clone git@github.com:ユーザー名/リポジトリ名.git

cloneするときは必ずSSHのURLを使ってください。HTTPSにしてしまった場合は以下で変更できます:

git remote set-url origin git@github.com:ユーザー名/リポジトリ名.git

現在のリモートURLを確認:

git remote -v

6. 基本の流れ: add / commit / push / pull

Gitの基本操作は以下の4ステップです。

ファイルを変更したら:

# 1. 変更したファイルをステージング（コミット対象に追加）
git add ファイル名
# または全ファイルをまとめて追加
git add .

# 2. コミット（変更をローカルに記録）
git commit -m "変更内容の説明"

# 3. プッシュ（GitHubにアップロード）
git push

他のPCでの変更を取り込むとき:

# リモートの変更をローカルに取り込む
git pull

よく使うコマンド一覧

コマンド説明
git status現在の状態を確認（変更ファイル、ステージング状況）
git logコミット履歴を表示
git diff変更内容を確認
git add .全ての変更をステージングに追加
git commit -m "メッセージ"変更をコミット
git pushリモートにプッシュ
git pullリモートの変更を取り込む
git pull --rebaseリモートの変更をrebaseで取り込む
git fetchリモートの情報を更新（マージはしない）
git remote -vリモートURL確認

7. ブランチ操作

ブランチは「別バージョン」を作るための仕組みです。元のコードに影響を与えずに新しい機能を開発したり実験したりできます。

ブランチの仕組み

ブランチの正体はポインタです。コミット履歴上のあるコミットを指しているだけです。

  • git fetch -- リモートのコミット履歴をローカルにアップデートし、origin/<ブランチ名> にリモートのポインタ情報が入る
  • git merge origin/<ブランチ名> -- その情報を今いるブランチに取り込む
  • git pull = git fetch + git merge

基本操作

# ブランチ一覧を表示
git branch

# 新しいブランチを作成して切り替え
git checkout -b 新しいブランチ名

# ブランチを切り替え
git checkout ブランチ名

# 新しいブランチをリモートにプッシュ（初回）
git push -u origin ブランチ名
# 以降は git push のみでOK

# ブランチを統合（mainにいる状態で）
git merge 統合したいブランチ名

# ブランチを削除（ローカル）
git branch -d ブランチ名

# ブランチを削除（リモート）
git push origin --delete ブランチ名

リモートのブランチをローカルに取り込む

git fetch
git checkout -b ローカルブランチ名 origin/リモートブランチ名

これでローカルブランチがリモートブランチと同じコミットを指すようになります。以降は git pull や git push が自動的に対応します。

merge と rebase の違い

mergerebase
やること2つのブランチを合流コミットを付け替え
履歴分岐が残る（安全）一直線になる（きれい）
向いてる用途共有ブランチ個人ブランチ
注意点merge commitが作られるコミットIDが変わるため共有後は危険

8. 特定のコミットに戻したいとき

まず git log でコミットIDを確認します。

1. そのコミットのコードを見たいだけ

git checkout <コミットID>

最新に戻るには git checkout <ブランチ名>。プログラムだけコピペしたいならこれが一番お手軽です。

2. そのコミットに戻して、それ以降を削除

git reset --hard <コミットID>

取り消せないので注意。確信があるときだけ使ってください。

3. そのコミットに戻して、履歴も残す

git revert <IDn> <IDn-1> ... <ID1>

そのコミットより後のIDに対して逆順で revert します。履歴が残るので安全です。

9. コンフリクトの解決

複数の場所から同じファイルを編集すると、コンフリクト（衝突）が起きることがあります。

rebase で解決する場合

git pull --rebase
git status              # コンフリクトファイルを確認
# エディタでファイルを開いてコンフリクト箇所を修正
git add .
git rebase --continue
git push

merge の場合は rebase の部分を merge に読み替えてください。git pull の代わりに git fetch → git rebase origin/<ブランチ名> でもOKです。

リモートが完全に正しい場合

git reset --hard origin/<ブランチ名>

ファイルごとに選択する場合

# リモートが正しいファイル
git checkout --theirs <ファイル名>

# ローカルが正しいファイル
git checkout --ours <ファイル名>

ローカルが正しくてリモートを上書きしたい場合

git push -f origin <ブランチ名>

強制プッシュはリモートの履歴を上書きするので、自分だけのブランチでのみ使ってください。

SSH / Remote-SSH

自分のPCでプログラムを実行するとスペック不足でフリーズすることがあります。SSH（Secure Shell）はネットワークを通じて他のPCに安全に接続し、遠隔操作する技術です。研究室に行かなくても、自宅から研究室のPCを操作できます。

VSCodeのRemote-SSH拡張機能を使えば、自分のPCのVSCode上で研究室PCの環境でコーディングできます。ファイルもターミナルもリモート側のものを直接操作でき、あたかも研究室のPCの前にいるかのように作業できます。

SSHの仕組み

SSHは「公開鍵暗号方式」を使って安全に通信します。

  1. 秘密鍵（自分のPCに保存、絶対に人に見せない）と公開鍵（接続先サーバに置く）のペアを作る
  2. 接続時、サーバが公開鍵で暗号化したデータを送り、クライアントが秘密鍵で復号して認証する
  3. パスワード不要で安全に接続できる

ネットワーク接続: Tailscaleの導入

研究室のPCに外部から接続するには、同じネットワーク上にいる必要があります。TailscaleはVPNを簡単に構築できるツールで、異なるネットワークにあるPC同士を仮想的に同じネットワークに接続できます。

  1. 各PCでTailscaleをインストールし、同じアカウントでログイン
  2. Tailscaleを起動する
    • Windowsではアプリを開いて起動
    • Ubuntuでは sudo tailscale up
  3. Tailscaleの管理画面からSSHしたいサーバのIPアドレスを確認

接続テスト:

ssh ユーザー名@IPアドレス
# 例: ssh andolab@100.94.152.23

SSH鍵の設定（公開鍵認証）

毎回パスワードを入力しなくて済むように、SSH鍵ペアを設定します。

Step 1: 鍵ペアを作成する

鍵の名前に接続先サーバ名を入れておくとわかりやすいです。

ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_andolab

Step 2: 公開鍵をサーバに送信する

ssh-copy-id -i ~/.ssh/id_ed25519_andolab.pub ユーザー名@IPアドレス

これでサーバの ~/.ssh/authorized_keys に公開鍵が自動的に追加されます。

Step 3: SSH設定ファイルを書く

~/.ssh/config に以下を追記します:

Host andolab
    HostName 100.94.152.23
    User andolab
    IdentitiesOnly yes
    IdentityFile ~/.ssh/id_ed25519_andolab

これで ssh andolab だけで接続できるようになります。

Step 4: 接続テスト

ssh andolab

パスワードなしで接続できれば成功です。

VSCode Remote-SSH の設定

VSCodeの拡張機能「Remote - SSH」をインストールすると、VSCode上からSSH接続先のファイルを直接編集できるようになります。

  1. VSCodeの拡張機能から「Remote - SSH」をインストール
  2. コマンドパレット（Ctrl + Shift + P）で「Remote-SSH: Connect to Host...」を選択
  3. 先ほど ~/.ssh/config に設定したホスト名（例: andolab）を選択
  4. 接続が完了すると、VSCodeのファイルエクスプローラーやターミナルがリモートPCのものになる

VSCodeはWindowsのソフトウェアですが、WSL上のUbuntuやリモートサーバで開けるのは、裏でサーバプロセスを立てて接続しているからです。

tmux

SSHして実験を開始しても、自分のPCがスリープした瞬間にSSHセッションが切断され、実行中のシェルスクリプトも停止してしまいます。それを防ぐのがtmuxです。

tmuxは1つのターミナル上で複数の仮想端末を管理できるツールで、SSH接続が切れてもプログラムを実行し続けられます。

  • SSH接続が切れてもプログラムを実行し続けられる
  • 複数の作業画面（タブのようなもの）を切り替えられる
  • 画面を分割して同時に複数の作業ができる
  • 作業状態を保持したまま離脱（detach）し、後で再接続できる

3階層構造

概念説明
セッション作業単位の一番大きな枠。SSHを切断しても残る。最も重要な概念
ウィンドウタブのようなもの。用途ごとに分ける（run, log, monitorなど）
ペインウィンドウ内を上下・左右に分割した画面。長時間実行だけなら不要

基本コマンド

tmuxの操作はプレフィックスキー（デフォルトは Ctrl + B）の後にキーを押す形式です。

セッション操作（ターミナルから実行）

コマンド説明
tmux new -s セッション名名前付きセッションを作成（推奨）
tmux lsセッション一覧を表示
tmux a -t セッション名セッションに再接続（attach）
tmux kill-session -t セッション名セッションを完全に終了

tmux内の操作（プレフィックス + キー）

キー説明
Ctrl+B → dセッションからデタッチ（離脱）
Ctrl+B → c新しいウィンドウを作成
Ctrl+B → n次のウィンドウへ移動
Ctrl+B → p前のウィンドウへ移動
Ctrl+B → 0-9番号指定でウィンドウ移動
Ctrl+B → ,ウィンドウ名を変更
Ctrl+B → &ウィンドウを閉じる

設定ファイル

設定ファイルの場所: ~/.tmux.conf（最初は存在しないことが多いので、自分で作成します）

例: プレフィックスキーを Ctrl+B から Ctrl+A に変更する:

# ~/.tmux.conf
unbind C-b
set -g prefix C-a
bind C-a send-prefix

設定を反映:

tmux source-file ~/.tmux.conf

長時間処理の実行方法（鉄板手順）

大前提: tmuxは「SSH切断に強い」だけです。ローカルPCをスリープしても処理が続くわけではありません。長時間処理は必ず「スリープしないマシン（サーバ）」上で実行してください。

実行手順

# 1. サーバにSSH接続
ssh andolab

# 2. tmuxセッションを作成
tmux new -s experiment

# 3. 実験スクリプトを実行
bash run_experiment.sh

# 4. デタッチ（Ctrl+B → d）
#    ※ exitは絶対にしない！

再接続

# サーバにSSH接続
ssh andolab

# セッションに再接続
tmux a -t experiment

操作ごとの結果まとめ

操作結果
Ctrl+B → d（デタッチ）セッションは残る（安全）
exitそのウィンドウのbashが終了。最後のウィンドウならセッションも終了
SSH切断tmuxがあれば問題なし
ローカルPCスリープサーバ上なら処理は継続。ローカルなら停止

まとめ: tmuxは「切断に強い作業場」。長時間処理は「スリープしないマシン」で。exitせずdetachが正解。

tgif / gnuplot

安藤研の卒論・報告書では、図の作成にtgif（ベクター図形の描画）とgnuplot（グラフの描画）を使います。どちらもEPS形式で出力し、LaTeXに貼り付けます。

tgif

tgifはLinux上で動作するベクター描画ツールです。グラフ理論の図（頂点・辺・ラベルなど）を描くのに適しています。

インストール

sudo apt install tgif

基本操作

  • tgif で起動。GUIが開く
  • 左のツールバーから丸（頂点）、線（辺）、テキスト（ラベル）などを選んで描画
  • 保存すると .obj ファイルができる
  • EPS出力: メニューの File → Print → EPS、または以下のコマンド: 
    tgif -print -eps ファイル名.obj

よく使う操作

操作説明
丸を描くOvalツールで描画。Shiftを押しながらで正円
線を描くPolyツールで頂点間を結ぶ
テキスト追加Textツールでクリック。数式は TeX 記法が使える
オブジェクトの移動Selectツールでドラッグ
コピー選択して Ctrl+C → Ctrl+V
整列複数選択して Align メニューで整列

tgifはリモートSSH先のLinuxマシンで使う場合、X11転送が必要です: ssh -X andolab で接続してから tgif を起動します。VSCode Remote-SSHではGUIが使えないので注意。

gnuplot

gnuplotはデータをグラフにプロットするツールです。実験結果の可視化に使います。

インストール

sudo apt install gnuplot

基本的な使い方

プロットスクリプト（.gp ファイル）を書いて実行します。

スクリプトの例

以下は実験データをEPSファイルに出力する例です。

# plot.gp
# 出力形式（EPS）
set terminal postscript eps enhanced color font "Times-Roman,20"
set output "result.eps"

# サイズと余白
set size ratio 0
set lmargin 8
set rmargin 2
set tmargin 2
set bmargin 4

# 軸ラベル
set xlabel "time [sec]"
set ylabel "function value"

# グリッド
set grid xtics ytics

# y軸目盛り（整数）
set format y "%.0f"

# プロット
plot "data.dat" using 1:2 with points pt 7 ps 1.4 lc rgb "blue" title "result"

実行方法

gnuplot plot.gp

これで result.eps が生成されます。

データファイルの形式

.dat ファイルはスペース区切りのテキストです。

# data.dat
# x  y
1  10
2  15
3  12
4  20
5  18

複数データの重ね描き

plot "data1.dat" using 1:2 with points pt 7 ps 1.4 lc rgb "red" title "method A", \
     "data2.dat" using 1:2 with points pt 7 ps 1.4 lc rgb "blue" title "method B"

よく使うオプション

オプション説明
with points散布図
with lines折れ線グラフ
with linespoints線＋点
pt 7点の形（7 = 塗りつぶし丸）
ps 1.4点のサイズ
lc rgb "blue"色の指定
set key right top凡例の位置
set xrange [0:10]x軸の範囲指定

LaTeXへの貼り付け方

tgifやgnuplotで生成したEPSファイルは、LaTeXの figure 環境で挿入します。

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.8\linewidth]{tgifs/graph.eps}
  \caption{グラフ $G$ の構造.}
  \label{fig:graph}
\end{figure}

ファイル配置のおすすめ

卒論リポジトリでは以下のようにフォルダ分けすると管理しやすいです。

sotsuron/main/
  sotsuron_main_suganuma.tex
  tgifs/          ← tgifで作った図（.obj と .eps）
  gnuplots/       ← gnuplotのスクリプトとデータ（.gp, .dat, .eps）

EPSファイルについて

EPS（Encapsulated PostScript）はベクター形式の画像です。拡大しても劣化しないため、論文の図に適しています。LaTeXで使うには \usepackage[dvipdfmx]{graphicx} が必要です。

AI

近年のAIエージェントは急速に進化しており、プロジェクト内のファイルを読み込んで文脈を理解したうえで、文章やプログラムの作成・実行・デバッグ、Webからの情報収集、他のアプリとの連携まで行うことができます。

本ページで紹介したGit・SSH・tmuxの設定なども、AIエージェントに依頼すれば自動的に進めてくれます。

おすすめ: Claude Code

筆者が使っているのはClaude Code（Anthropic社）です。ターミナル上で動作するAIエージェントで、以下のようなことができます:

  • プロジェクト内のファイルを読み込んで文脈を理解する
  • コードを書く・修正する・デバッグする
  • シェルコマンドを実行する
  • Webから情報を検索する
  • Git操作やファイル管理を自動化する

研究で使うくらいならProプランで十分です。無料のAIエージェントがあれば、まずはそれを使ってみるのもいいでしょう。

AIとの付き合い方

学習の流れ（推奨）

  1. AIにやり方・理由・不明点を聞く -- ここは「不確実でもいい場所」
  2. 実際に試す -- 環境・バージョン・自分の手で確認
  3. うまくいったこと・腑に落ちたことだけをノートに残す -- ノートは「検証済みの事実の集合」

AIの出力を鵜呑みにせず、自分で検証する癖をつけましょう。AIは強力なツールですが、最終的に判断するのは自分です。
